Kết nối một tài khoản hoặc tổ chức GitHub một lần trong **Cài đặt → GitHub**. Sau đó, bất kỳ pull request nào có tên nhánh, tiêu đề hoặc phần thân chứa mã issue (ví dụ `MUL-123`) được **tự liên kết** vào [issue](/issues) đó, xuất hiện dưới mục **Pull requests** ở thanh bên của issue, và — khi PR được merge — chuyển issue sang **Done**.

Không cần thiết lập theo từng issue. Toàn bộ luồng chạy theo mã định danh.

## Tích hợp làm gì

| Bề mặt | Hành vi |
|---|---|
| **Cài đặt → GitHub** | Admin của không gian làm việc thấy tab GitHub với công tắc tổng, nút **Connect GitHub** và các công tắc tính năng (thanh bên PR, Co-authored-by, tự liên kết). Cài xong bạn được đưa về tab GitHub. |
| **Thanh bên issue → Pull requests** | Mọi PR tự liên kết vào issue này, kèm tiêu đề, repo, trạng thái (`Open` / `Draft` / `Merged` / `Closed`) và tác giả. Bấm một dòng để nhảy tới PR trên GitHub. |
| **Webhook (nền)** | Với mỗi sự kiện `pull_request`, Multica upsert dòng PR, quét PR tìm mã issue và dựng (lại) các dòng liên kết. Idempotent — phát lại một lượt giao nhận là no-op. |
| **Tự đổi trạng thái khi merge** | Khi PR chuyển sang `merged`, mọi issue liên kết chưa ở `Done` hoặc `Cancelled` được chuyển sang `Done`. Thay đổi trạng thái được ghi vào dòng thời gian với nguồn `github_pr_merged`. |

Chỉ bản thân PR được phản chiếu. Commit, ref nhánh không có PR mở, và trạng thái CI check **không** được mô hình hóa. Tích hợp chủ ý giữ hẹp.

## Mã định danh được khớp thế nào

Webhook trích mã từ ba trường, theo thứ tự: **nhánh head của PR**, **tiêu đề PR**, **thân PR**. Bộ khớp:

- Không phân biệt hoa thường — `mul-123`, `MUL-123`, `Mul-123` đều khớp.
- Có biên — `\b` bên trái và neo chữ số bên phải giữ nó khỏi vồ số phiên bản như `v1.2-3` hay chuỗi kiểu email.
- Theo phạm vi không gian làm việc — chỉ khớp [tiền tố issue](/workspaces) của chính không gian làm việc. `FOO-1` trong không gian làm việc có tiền tố `MUL` bị bỏ qua, kể cả khi phần số trùng một issue khác.
- Khử trùng lặp — ghi `MUL-1, MUL-1` trong thân chỉ liên kết issue một lần.

Bạn có thể nhắc **nhiều issue** trong một PR. `Closes MUL-1, MUL-2` liên kết PR với cả hai, và merge nó đưa cả hai lên `Done`.

## Quy tắc tự merge-thành-Done

Khi trường `merged` của PR lật sang `true`, mọi issue liên kết được đánh giá:

| Trạng thái hiện tại của issue | Kết quả |
|---|---|
| `done` | Không đổi (đã kết thúc). |
| `cancelled` | **Không đổi** — cancelled nghĩa là người dùng đã chủ động bỏ việc; tích hợp không ghi đè tín hiệu đó. |
| Còn lại (`todo`, `in_progress`, `in_review`, `blocked`, `backlog`) | Chuyển sang `done`. |

Đóng PR **mà không** merge chỉ cập nhật trạng thái thẻ PR thành `Closed`. Các issue liên kết đứng nguyên — người dùng là người quyết định đóng-không-merge nghĩa là gì.

<Callout type="info">
Hành động được ghi cho actor `system` trên dòng thời gian. Người theo dõi issue nhận thông báo hộp thư cho thay đổi trạng thái, y như khi một con người chuyển nó.
</Callout>

## Những gì không tự liên kết

- **Mã trong thông điệp commit** — chỉ nhánh / tiêu đề / thân được quét. Commit tên `MUL-123: fix login` không tự liên kết trừ khi chuỗi đó cũng xuất hiện trong tiêu đề hoặc thân PR.
- **Mã trong bình luận của PR** — chỉ metadata của chính PR được quét; các bình luận GitHub về sau bị bỏ qua.
- **PR trong repo chưa cài App** — không có App, Multica không bao giờ nhận webhook.
- **Liên kết PR vào issue thủ công** — chưa có UI cho việc này. Nếu quy ước của nhóm đặt mã ở chỗ Multica không đọc, thêm nó vào tiêu đề hoặc thân PR.

## Ngắt kết nối

Trong **Cài đặt → GitHub** không có danh sách installation — bạn quản lý các installation sẵn có từ GitHub trực tiếp:

- **Từ GitHub** — gỡ Multica GitHub App tại `https://github.com/settings/installations` (cá nhân) hoặc `https://github.com/organizations/<org>/settings/installations` (tổ chức). Multica nhận webhook `installation.deleted` và xóa dòng theo thời gian thực; tab Cài đặt đang mở cập nhật không cần tải lại.
- **Ngắt từ trong Multica chỉ dành cho admin** — nút Disconnect trên tab GitHub bị ẩn với người không phải admin. Nó vẫn khả dụng kể cả khi công tắc tổng GitHub đang tắt, để admin vẫn thu hồi được installation cũ sau khi tắt nhanh tính năng.

Sau khi ngắt, các dòng PR đã phản chiếu vẫn nằm trong cơ sở dữ liệu để thanh bên issue lịch sử vẫn hiện những gì từng liên kết, nhưng sự kiện webhook mới từ installation đó không được chấp nhận nữa.

## Quyền và khả năng nhìn thấy

- **Kết nối / ngắt kết nối** cần **owner hoặc admin** của không gian làm việc. Member thấy mô tả thẻ nhưng không có nút Connect.
- Thanh bên **Pull requests** trên issue hiện với bất kỳ ai đọc được issue — cùng quyền với phần còn lại của chi tiết issue.
- GitHub App xin quyền **chỉ đọc** với pull request và metadata. Multica không bao giờ đẩy commit, bình luận hay status check ngược về GitHub.

## Thiết lập tự vận hành

Nếu bạn dùng Multica Cloud, tích hợp đã cấu hình sẵn — bỏ qua mục này.

Với tự vận hành, bạn tạo một GitHub App, trỏ nó về server của mình và đặt hai biến môi trường. Toàn bộ luồng bên dưới.

### 1. Tạo GitHub App

Vào một trong hai:

- Tài khoản cá nhân → `https://github.com/settings/apps/new`
- Tổ chức → `https://github.com/organizations/<org>/settings/apps/new`

Điền:

| Trường | Giá trị |
|---|---|
| **GitHub App name** | Gì đó dễ nhận, ví dụ `Multica` hoặc `Multica (staging)`. |
| **Homepage URL** | Frontend Multica của bạn, ví dụ `https://multica.example.com`. |
| **Callback URL** | Để trống — Multica không dùng danh tính OAuth. |
| **Setup URL** | `https://<api-host>/api/github/setup`. **Tích "Redirect on update"**. |
| **Webhook → Active** | Bật. |
| **Webhook URL** | `https://<api-host>/api/webhooks/github`. |
| **Webhook secret** | Sinh một chuỗi ngẫu nhiên dài (ví dụ `openssl rand -hex 32`). Bạn sẽ dán đúng giá trị này vào env của Multica ở bước 2. |
| **Permissions → Repository → Pull requests** | **Read-only**. |
| **Permissions → Repository → Metadata** | Read-only (bắt buộc). |
| **Subscribe to events** | Tích **Pull request**. |
| **Where can this GitHub App be installed?** | Tùy bạn. `Only on this account` là ổn cho thiết lập một tổ chức. |

Sau **Create GitHub App**, ghi lại hai thứ từ trang chi tiết App:

- **Public link** ở trên cùng — đuôi của nó là slug. `https://github.com/apps/multica-acme` → slug = `multica-acme`.
- **Webhook secret** vừa sinh (về sau không đọc lại được từ GitHub — lưu ngay).

<Callout type="warning">
**Webhook secret ≠ Client secret.** Trang cài đặt App xếp hai trường này cạnh nhau. **Webhook secret** là thứ ký payload `pull_request` — đó là cái Multica cần. **Client secret** dành cho OAuth và tích hợp này không dùng. Nhầm hai cái sẽ sinh `401 invalid signature` khó hiểu ở mọi lượt giao webhook.
</Callout>

### 2. Đặt biến môi trường

Trên API server:

```dotenv
GITHUB_APP_SLUG=multica-acme
GITHUB_WEBHOOK_SECRET=<webhook secret bạn đã sinh>

# Tùy chọn nhưng nên có — điền tên tài khoản kết nối ngay khi cài
# thay vì chờ webhook đầu tiên refresh:
GITHUB_APP_ID=<App ID dạng số từ trang cài đặt App>
GITHUB_APP_PRIVATE_KEY=<nguyên khối PEM, gồm cả dòng BEGIN/END>
```

`GITHUB_APP_SLUG` và `GITHUB_WEBHOOK_SECRET` là bắt buộc. Thiếu một trong hai:

- Nút `Connect GitHub` trong Cài đặt bị **vô hiệu** và hiện gợi ý "not configured".
- Endpoint `/api/webhooks/github` trả **`503 github webhooks not configured`** — Multica từ chối xử lý sự kiện khi không có secret, thay vì lặng lẽ coi mọi chữ ký là hợp lệ.

`GITHUB_APP_ID` và `GITHUB_APP_PRIVATE_KEY` là **tùy chọn**. Chúng cho phép callback thiết lập gọi endpoint `/app/installations/{id}` xác thực bằng App của GitHub để lấy tên tổ chức hoặc người dùng thật ngay lúc cài. Không có chúng, thẻ kết nối hiện tạm `Connected to unknown` cho tới khi GitHub giao webhook `installation.created` (thường trong vài giây), lúc đó Multica refresh dòng và phát cập nhật realtime để tab Cài đặt đang mở tự cập nhật. Sinh private key ở **Private keys → Generate a private key** trên trang cài đặt App; dán nguyên khối PEM (gồm cả dòng `-----BEGIN/END RSA PRIVATE KEY-----`) vào biến env, giữ nguyên xuống dòng.

`FRONTEND_ORIGIN` cũng phải được đặt (mọi bản tự vận hành production đều đã có); callback thiết lập đưa người dùng về `<FRONTEND_ORIGIN>/settings?tab=github` sau khi cài.

Khởi động lại API sau khi đặt biến env.

### 3. Chạy migration

Tích hợp mang bảng của nó trong migration `079_github_integration`. Nếu bạn nâng cấp từ bản cũ:

```bash
make migrate-up
```

Ba bảng được tạo: `github_installation`, `github_pull_request`, `issue_pull_request`. Chúng cascade-delete cùng không gian làm việc, nên xóa không gian làm việc tự dọn chúng.

### 4. Kết nối từ UI

Trong Multica:

1. Mở **Cài đặt → GitHub** với vai trò owner hoặc admin.
2. Bấm **Connect GitHub**. GitHub mở trong tab mới.
3. Chọn repository muốn cấp quyền và **Install**.
4. GitHub chuyển hướng về `<api-host>/api/github/setup`, nơi ghi lại installation và đưa bạn về `<FRONTEND_ORIGIN>/settings?tab=github&github_connected=1`.

Sau đó, mở bất kỳ PR nào có nhánh / tiêu đề / thân chứa mã issue — trong vài giây, khối Pull requests xuất hiện trên trang chi tiết issue đó.

### 5. Kiểm chứng bằng curl

Nếu trang **Recent Deliveries** của GitHub báo `401 invalid signature` sau khi cài, hai bên đang khác secret. Cách nhanh nhất để biết bên nào sai là bỏ qua GitHub:

```bash
SECRET="<giá trị bạn đặt vào GITHUB_WEBHOOK_SECRET>"
BODY='{"zen":"test"}'
SIG=$(printf '%s' "$BODY" | openssl dgst -sha256 -hmac "$SECRET" -hex | awk '{print $NF}')
curl -i -X POST https://<api-host>/api/webhooks/github \
  -H "X-Hub-Signature-256: sha256=$SIG" \
  -H "X-GitHub-Event: ping" \
  -H "Content-Type: application/json" \
  -d "$BODY"
```

| Mã HTTP | Ý nghĩa | Cách sửa |
|---|---|---|
| `200` `{"ok":"pong"}` | Secret server đang nạp khớp `$SECRET` của bạn. Lệch nằm phía GitHub. | Sửa App → Webhook secret → **dán đúng giá trị** → **Save changes** (bấm ra ngoài ô mà không Save sẽ giữ secret cũ). Redeliver. |
| `401 invalid signature` | Secret server đang nạp **không** như bạn nghĩ. | Xác nhận biến env đã vào tiến trình đang chạy (ví dụ `kubectl exec` → `echo -n "$GITHUB_WEBHOOK_SECRET" \| wc -c`). Deploy lại. |
| `503 github webhooks not configured` | `GITHUB_WEBHOOK_SECRET` rỗng trong tiến trình. | Đặt biến env, khởi động lại API. |

## Hạn chế

Vài cạnh thô cần biết hôm nay:

- **Chưa có UI liên kết thủ công** — cách duy nhất để liên kết PR là có mã trong nhánh, tiêu đề hoặc thân của nó.
- **Không có trạng thái CI / check** — chỉ bản thân PR được phản chiếu. Trạng thái build, bình luận review và reviewer không hiện trong Multica.
- **Không có cấu hình mức không gian làm việc** cho quy tắc merge → Done — nó là mặc định cố định (`merged → done`, trừ `cancelled`). Ánh xạ tùy biến theo không gian làm việc là bổ sung tương lai.
- **Nhiều PR về một issue xử lý bảo thủ khi merge** — nếu hai PR cùng nhắc `MUL-123` và cái đầu merge, issue được chuyển `Done` ngay. Thay đổi tiếp theo để chờ mọi PR liên kết xong mới chuyển đang được làm.
